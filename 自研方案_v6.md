## 三、核心技术栈

| 层 | 技术 | 理由 |
|:---|:---|:---|
| **后端** | Python FastAPI | 你已有 Python 3.14，AI/LLM生态最好 |
| **前端** | Vue 3（Web） | 直接上Web，不做命令行 |
| **数据库** | SQLite（开发）→ PostgreSQL（生产） | 起步简单，后期扩展 |
| **LLM/剧本** | DeepSeek API（主） / Ollama qwen3:14b（备） | 你都有，混合使用 |
| **出图引擎** | **ComfyUI + Stable Diffusion**（云GPU） | 模型可插拔，用哪个模型出哪种风格，运行时不绑定 |
| **视频引擎** | **ComfyUI + HunyuanVideo / LTX-Video**（同一台云GPU） | 同一套API，出图后直接图生视频 |
| **ComfyUI集成** | Python 自写 ComfyUIClient（REST + WebSocket） | 比用通义/即梦API控制力强10倍 |
| **GPU机器** | AutoDL RTX 3090（¥1.32/时），社区ComfyUI镜像 | 需要时开机，不用就关 |
| **开发占位API** | 通义万相（免费额度） | 没租GPU时也能跑通代码 |
| **视频合成** | FFmpeg + MoviePy | 你已安 |
| **字幕** | FFmpeg drawtext / srt 叠加 | 轻量 |
| **部署** | Docker（腾讯云¥38/年，跑FastAPI + Vue + DB） | GPU和app服务器分离 |
| **管线调度** | 自写Pipeline类 | 单次跑1个任务，按服务器能力决定并行 |
| **断点续传** | checkpoint落盘，挂了解修从断点继续 | 按需实现 |

## 四、Agent 管线设计（核心）

每个 Agent 是一个独立的 Python 类，包含：

```
Agent 基类：
  - name: str              # Agent名称
  - model_provider: str    # LLM/API提供商
  - input_schema: dict     # 输入格式
  - output_schema: dict    # 输出格式
  - run(input) → output    # 执行入口
```

### 7 个 Agent 流程详解

#### Agent 1: 剧本生成

```
输入: 用户主题 / 网文文本 / 一句话
处理: 
  - 用 DeepSeek 解析输入
  - 提取: 故事梗概 / 人物列表(姓名/性格/外貌) / 场景列表 / 剧情主线
  - 拆分为单集短剧结构: 钩子→发展→反转→收尾
  - 每集500-1200字
输出: {
  "title": "剧名",
  "genre": "末世/甜宠/修仙...",
  "episodes": [{
    "episode": 1,
    "summary": "本集概要",
    "scenes": [{
      "scene_id": 1,
      "location": "废弃工厂",
      "description": "主角躲藏在生锈的机器后",
      "dialogue": "...",
      "mood": "紧张"
    }]
  }],
  "characters": [{"name": "林峰", "gender": "男", "appearance": "...", "personality": "..."}],
  "consistency_key": "林峰-consistency-6789"
}
```

#### Agent 2: 分镜规划

```
输入: 剧本(每集 scenes 列表)
处理:
  - 每个 scene → 一组分镜(shot)
  - 每个 shot 标注: 景别(远/中/近/特写) / 运镜(固定/推/拉/摇) / 时长(3-12s)
  - 提取SD提示词: "主体+动作+场景+光影+风格+画质"
输出: {
  "shots": [{
    "shot_id": 1,
    "scene_id": 1,
    "description": "林峰从废弃机器后探出头，警惕地望着前方",
    "shot_type": "中景",
    "camera_movement": "固定",
    "duration": 5,
    "character": "林峰",
    "character_ref": "林峰-consistency-6789",
    "sd_prompt": "一名年轻男子短发眼神坚毅，从生锈机器后探出头，废弃工厂昏暗光线，写实风格，8k",
    "sd_negative": "模糊,低质量,变形,多余手指",
    "video_motion": "镜头固定,角色轻微呼吸动作"
  }]
}
```

#### Agent 3: 角色定妆照

```
输入: 剧本中的 characters 列表
处理:
  - 通过 ComfyUI API 提交 SD workflow
  - 生成每个角色的定妆照（正面半身/全身）
  - 锁定 seed + ControlNet 参考图，保证后续一致
  - 开发者模式：没GPU时走通义万相API占位
输出: {
  "character_assets": [{
    "name": "林峰",
    "seed": 6789,
    "sd_model": "realisticVision-v51",
    "assets": {
      "portrait": "url_or_path",   # 头像
      "full_body": "url_or_path"   # 全身照
    },
    "controlnet_image": "url_or_path",  # 后续所有分镜用这个作为ControlNet参考
    "ip_adapter_image": "url_or_path"   # IP-Adapter参考
  }]
}
```

#### Agent 4: 出图（ComfyUI + SD）

```
输入: 分镜列表 + 角色资产
处理:
  - 每个 shot → 构建 ComfyUI workflow JSON
  - 关键：所有带"林峰"的分镜引用同一张 ControlNet 参考图 + seed
  - 批量提交所有 shots 到 ComfyUI API
  - WebSocket 实时监听进度
  - 下载所有输出图片
输出: {
  "images": [{
    "shot_id": 1,
    "image_path": "output/shot_001.png",
    "width": 1080,
    "height": 1920
  }]
}
```

**ComfyUI 工作流示意图（一个shot）：**
```
[Checkpoint Loader] → [CLIP Text Encode (Positive)] ─┐
  (realisticVision)    (sd_prompt)                    │
                                                     ├→ [KSampler] → [VAE Decode] → [Save Image]
[ControlNet Loader] → [ControlNet Apply] ────────────┘
  (角色参考图)        (控制姿态/构图)
                                          [CLIP Text Encode (Negative)]
                                            (sd_negative)

不同 shot 之间只替换: sd_prompt, seed, (某些shot替换ControlNet参考图)
```

#### Agent 5: 视频生成（ComfyUI + HunyuanVideo / LTX-Video）

```
输入: 分镜图片列表 + 每个shot的video_motion描述
处理:
  - 每个 shot 的图片 → 构建图生视频 workflow JSON
  - 提交到同一台 ComfyUI（已经装了HunyuanVideo节点）
  - 参数控制: 镜头推拉/角色动画/时长
输出: {
  "videos": [{
    "shot_id": 1,
    "video_path": "output/shot_001.mp4",
    "duration": 5.0,
    "fps": 24
  }]
}
```

**ComfyUI 图生视频工作流：**
```
[Load Image] → [HunyuanVideo Image to Video] → [VAE Decode] → [Save Video]
  (shot_001.png)    (video_motion prompt, 时长5s)
```

#### Agent 6: 字幕生成 + 视频合成

```
输入: 所有视频分段 + 剧本对话时间线
处理:
  - 生成 SRT 字幕文件（从剧本提取每句对话的时间戳）
  - FFmpeg 拼接所有视频分段
  - FFmpeg 叠加字幕到视频
输出: {
  "final_video": "output/剧名_E01_final.mp4",
  "duration": 180,
  "format": "mp4"
}
```

#### Agent 7: 发布（可选，后续扩展）

```
输入: 最终视频文件
处理:
  - 预留发布接口
  - 后续接抖音/快手/B站 API 自动发布
输出: 发布结果
```

## 五、核心引擎：ComfyUI 深度集成

### 5.1 为什么选 ComfyUI + SD，而不是云 API

| 维度 | 通义万相/即梦云API | **ComfyUI + SD + HunyuanVideo** |
|:---|:---|:---|
| **角色一致性** | 靠prompt硬调，不行 | ControlNet + IP-Adapter + LoRA，真正锁住角色 |
| **出图控制力** | 黑盒，给几个参数 | **全可控**: prompt/seed/CFG/LoRA/ControlNet全部编程调整 |
| **模型自由** | 模型固定 | 换任何模型: 写实/二次元/水墨/灵梦 |
| **视频控制力** | 黑盒 | LTX-Video/HunyuanVideo节点参数全控 |
| **工作流复用** | — | ComfyUI可视化拖出来→导出JSON→代码直接复用 |
| **成本** | 按次付费 | AutoDL ¥1.32/时，不开机不花钱 |
| **数据隐私** | 图片上传云 | 在自己GPU机器上，数据不外流 |
| **完全自主** | ❌ 依赖供应商 | ✅ 全开源可控 |

### 5.2 ComfyUIClient 设计

### 5.3 角色一致性实现

```
角色一致性 = IP-Adapter + ControlNet + 相同seed

Agent 3（定妆照）阶段：
  → 生成角色定妆照
  → 记录 seed + IP-Adapter参考图 + ControlNet姿态图

Agent 4（出图）阶段：
  → 每个涉及该角色的shot：
    1. Load checkpoint（同一模型）
    2. ControlNet: 载入角色定妆照作为参考 → ControlNetApply
    3. IP-Adapter: 载入角色定妆照作为风格/外观参考
    4. CLIP Text Encode: 用角色的sd_prompt
    5. KSampler: 固定seed = 角色consistency_seed + shot_id偏移
    6. 生成 → 输出图片

效果：同一角色在所有分镜中保持外貌/服装/气质一致
```

### 5.4 开发时的占位策略（Mock模式）

**通义万相占位已被否决。** 原因：
- prompt写法、长宽比、画风与SD完全不兼容
- 用通义调出来的prompt，放到ComfyUI+SD上90%会崩
- 看似"走通了"实则"白走通"

替代方案：Phase 0/1阶段涉及出图/视频的环节，直接使用 **MockProvider**（空实现返回假数据）。
等到Phase 2租了AutoDL，直接上ComfyUI真机调。

```
开发阶段示意图：

Agent 1（剧本） -> 真调DeepSeek OK
Agent 2（分镜） -> 真调DeepSeek OK
Agent 3（角色） -> MockProvider返回假图片路径 OK（只验证数据流）
Agent 4（出图） -> MockProvider返回假图片路径 OK
Agent 5（视频） -> MockProvider返回假视频路径 OK
Agent 6（字幕） -> FFmpeg真合成，用假图片占位 OK

等到Phase 2：将所有MockProvider一键切换为ComfyUIProvider
```

**优点：**
- 0 API费用
- Pipeline数据流完全可测
- 不受通义/即梦/任何第三方的API限制

### 5.5 环境锁定：ComfyUI Docker 镜像

**风险**：ComfyUI的节点版本、模型路径、自定义节点依赖在不同机器上可能完全不兼容。在AutoDL上调好的workflow.json，换一台机器可能直接报错 Node not found。

**应对**：把ComfyUI完整环境（含所有自定义节点、模型）打包成Docker镜像，上传到阿里云容器仓库。每次租GPU时拉取同一镜像。

```
# AutoDL 开机后：
docker pull registry.cn-hangzhou.aliyuncs.com/your-account/comfyui-env:v1
docker run --gpus all -p 8188:8188 registry.cn-hangzhou.aliyuncs.com/your-account/comfyui-env:v1
```

生产环境 = 同一镜像，零差异。

### 5.6 批量策略：异步回调队列 + 边生成边落盘

**问题**：100个视频分段串行要1.5-3小时。中途网络中断一次，全部重来。

**方案**：ComfyUIClient 内置 `asyncio.Queue` 异步回调队列。

```
batch_run_sd(shots):
  1. 启动后台消费者 _output_consumer (一直等待队列)
  2. 循环提交所有 shot 到 ComfyUI（不等待）
  3. 每提交一个，创建后台协程 _watch_and_enqueue
     -> WebSocket 监听该任务完成
     -> 完成后立即推入 _completed_queue
  4. 消费者从队列取出 -> 立即下载 -> 落盘到 storage/output/shot_{id}.png
  5. 触发进度回调：completed/total/shot_id
     -> 前端展示 "已完成 5/100"
  6. 废片自动检测 -> signal 给外层 retry 逻辑
```

**优点**：
- 3小时生成时间不必干等，前端边生成边预览
- 即使网络中断，已落盘的分段不丢失
- 断网恢复后只重跑未完成的分段（依赖checkpoint记录已完成列表）
- 用户看到 "已完成 5/100" 就有信心继续等

batch_run_video() 使用完全相同的模式。

## 六、关键系统设计

### 6.1 Provider 分层

```
统一的 Provider 抽象层，后面想加啥加啥：

  class LLMProvider(ABC):        # 文本生成
    async def generate(prompt, model) → str

  class ImageProvider(ABC):      # 图片生成
    async def generate(prompt, ref_image, seed) → str

  class VideoProvider(ABC):      # 视频生成
    async def generate(input_image, prompt, duration) → str

  → ComfySDImageProvider(ImageProvider):    # ComfyUI + SD 出图
  → ComfyHunyuanVideoProvider(VideoProvider): # ComfyUI + HunyuanVideo
  → MockImageProvider(ImageProvider):       # Mock占位（开发阶段，返回假路径）
  → MockVideoProvider(VideoProvider):       # Mock占位（开发阶段，返回假路径）
  → DeepSeekLLMProvider(LLMProvider):       # DeepSeek 写剧本
  → OllamaLLMProvider(LLMProvider):         # Ollama 本地（备用）

  # 预留接口：
  → TextToVideoProvider(ABC):   # 文生视频（未来扩展）
```

### 6.2 双轨制：资产生产 + 单集制作

**Phase A（资产生产）：** 先批量生成所有角色定妆照 + 场景背景图 + 道具 -> 入库
**Phase B（单集制作）：** 每集调用资产库，只生成该集特有的分镜

角色入库标记：Character 模型 `is_asset_library` 字段。如果角色已有定妆照入库，Agent 3 直接跳过，复用之前的 ControlNet 参考图。

多集漫剧不用重复生成同一角色的定妆照。

### 6.3 人工干预节点

纯自动化管线的问题是一旦中间某步出错，整条线就断了。增加两个确认点：

1. **分镜确认**：Agent 2 生成分镜后，用户可手动调整再进入出图
2. **角色定妆照确认**：Agent 3 生成角色后，用户确认再用 ControlNet

### 6.4 风格配置文件

切换风格时自动切换模型+workflow+参数组：

```yaml
style_profiles:
  realistic:
    sd_model: realisticVision-v51
    sd_workflow: sd_shot_realistic.json
    video_workflow: img2video_realistic.json
  anime:
    sd_model: anything-v5
    sd_workflow: sd_shot_anime.json
    video_workflow: img2video_anime.json
  ink_wash:
    sd_model: base_model + 水墨LoRA
    sd_workflow: sd_shot_ink.json
    video_workflow: img2video_ink.json
```

用户选择风格后，系统自动切换对应的模型和工作流。

### 6.5 分镜校验层

Agent 2（分镜Agent）输出后在管线中自动校验：
- shot_type 是否在预定义集合内（远/中/近/特写/俯拍/仰拍）
- duration 是否在 3-12s 范围内
- sd_prompt 是否包含必要元素（主体+动作+场景+光影+画质）
- character_name 是否在已注册的角色列表中

不合格直接要求 LLM 重生成，重试上限3次。

### 6.6 管线调度

```
Pipeline 调度器：
  - 每个 Agent 是异步可插拔的
  - 支持串行/并行切换
  - 出错自动重试 + 废片自动检测重跑
  - 支持断点续传 + 每30分钟自动快照

调度器接口:
  class Pipeline:
    agents: List[Agent]
    state: PipelineState

    async def run(self, input: Dict) -> PipelineResult:
      for agent in self.agents:
        try:
          result = await agent.run_with_retry(input)
          state.save_checkpoint(agent.name, result)
        except Exception as e:
          state.mark_failed(agent.name, str(e))
      return PipelineResult(state.final_output())
```

### 6.7 存储设计

```
开发阶段:
  - SQLite（一个文件搞定所有）
  - 本地文件存储图片/视频
  - ComfyUI结果通过异步回调队列自动下载到 storage/output/

生产阶段:
  - PostgreSQL（腾讯云/阿里云 RDS）
  - 对象存储（阿里云OSS / 腾讯云COS）放图片和视频
  - ComfyUI结果自动转存OSS

主要表:
  stories          # 故事主表
  episodes         # 剧集
  shots            # 分镜
  characters       # 角色（含 is_asset_library 字段标记入库状态）
  character_assets # 角色定妆照资产（ControlNet参考图）
  pipeline_jobs    # 管线执行记录（含 checkpoint）
  published        # 发布记录
```
