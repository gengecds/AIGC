## 风险逐一推演

### 🔴 风险1：角色一致性（最关键）

**问题**：漫剧最核心的体验就是"角色不能变脸"。同一角色在分镜1和分镜20长不一样，就是废片。

现在方案：
```
Agent 3（定妆照）
  → 生成角色图片 + 记录 seed + IP-Adapter 参考图
  
Agent 4（出图）
  → 每个分镜带上 ControlNet（参考定妆照）+ IP-Adapter + 相同种子族
```

**到底能不能锁定？**

| 方案 | 一致性程度 | 复杂度 |
|:---|:---:|:---:|
| ① 纯 Prompt + 相同 seed | ❌ 不稳定，模型波动大 | 低 |
| ② Prompt + seed + LoRA | ⚠️ 需要每角色训练 LoRA（20分钟/个） | 中 |
| ③ **Prompt + seed + IP-Adapter** | ✅ 头像风格都能复现 | 中 |
| ④ **Prompt + seed + IP-Adapter + ControlNet** | ✅✅ 构图+外貌双重锁死 | 中高 |
| ⑤ ③ + ④ + InstantID/IPAdapter-FaceID | ✅✅✅ 适合特写镜头 | 高 |

**推荐**：方案④为主 + 方案⑤补充特写镜头

**但需要确认**：你"漫剧"的画面风格是什么？
- A. 写实风格（真人脸） → 用 RealisticVision + IP-Adapter-FaceID
- B. 二次元/动漫风格 → 用 Anything/NovelAI + IP-Adapter
- C. 水墨/特殊风格 → 需要特定模型 + 对应ControlNet

**风格不同，模型不同，这会影响所有后续选择。**

---

### 🟡 风险2：ComfyUI 批量出图速度

假设一部漫剧 5 集，每集 20 个分镜 = **100 个分镜**。

| 操作 | 每次耗时（RTX 3090） | 100次总耗时 |
|:---|:---:|:---:|
| SD 出图（1024×576，30步） | ~3-5秒 | **5-8分钟** |
| HunyuanVideo 图生视频（5秒片段） | ~60-120秒 | **100-200分钟** |

**卡点**：视频生成才是瓶颈。100个分镜的视频生成要1.5-3小时（串行）。

**应对**：
1. ComfyUI 支持并行排队（队列机制），多任务并行能缩短
2. 开发阶段先跑1集验证，不追求全量
3. 必要时多租1块卡并行（AutoDL ¥2.64/时）

**需要确认**：你能接受的单集漫剧生成时间？

---

### 🟡 风险3：网文解析质量

**问题**：用户输入可能是一段网文/一句话/一个大纲。DeepSeek 解析复杂网文能不能准确提取：

- ✅ 人物列表（姓名+性格+外貌）
- ✅ 场景列表
- ✅ 剧情主线（起承转合）
- ⚠️ 关键道具/设定（如"修真境界""武器名称"）
- ⚠️ 对话中的情感变化

**风险**：网文动辄几万字，DeepSeek 上下文够但解析准确度取决于 prompt 工程。

**应对**：分步解析 + 每步都做信息抽取，而不是一次生成全部。

---

### 🟡 风险4：视频质量

**问题**：HunyuanVideo / LTX-Video 图生视频的**成品样片**是什么效果？

目前没有实测，纯靠看评测。需要回答：
- 图生视频的动效幅度有多大？（角色眨眼？头发飘动？还是整个动作？）
- 5秒片段里角色能不能做复杂动作？（挥手、转身、说话口型？）
- 多个角色在同一画面时，动效是否分布合理？

**应对**：直接租1小时AutoDL，跑一段样片出来看。这是测试成本最低、能最快发现问题的方法。

---

### 🟢 风险5：字幕生成

**问题**：你说不要配音，但字幕仍然需要。字幕来自剧本对话，每个分镜对应的对话时间轴怎么确定？

**方案**：
```
剧本 → 每句对话绑定到 shot_id
每个 shot 的 video 片段时长已知（比如5秒）
→ 时间轴自动算出来：shot_1=0~5s, shot_2=5~10s, shot_3=10~15s
→ 对话插入对应的 SRT 时间戳
→ FFmpeg drawtext 叠加
```

**结论**：无风险。但你**字幕要什么样式**？（白字黑底？底部居中？字体大小？）

---

### 🟡 风险6：管线稳定性

**问题**：7 个 Agent 串行跑，中间任何一个环节挂了（API错误/网络问题/模型报错），前面的工作就全废了。

**应对方案**：
```
每个 Agent 完成后，结果落盘（JSON + 文件）
下一个 Agent 从磁盘读取上一步的结果

断点续传：
  Pipeline 跑在第3步挂了
  → 修好错误
  → 直接从第3步继续（不重跑1-2步）
```

**需要确认**：你认为断点续传是"必要的"还是"后面再补"？

---

### 风险7：项目定位

**问题**：这个项目最终做成什么？

| 定位 | 复杂度 | 你的目标 |
|:---|:---:|:---|
| 🛠️ **命令行工具** | 低 | 初期最合适的形态 |
| 🌐 **Web应用**（FastAPI + 前端） | 中 | 需要UI时再加 |
| 🔄 **全自动生产线**（接微信/定时/批量） | 高 | 未来扩展方向 |

**需要确认**：目前先做命令行工具？还是有Web需求？
